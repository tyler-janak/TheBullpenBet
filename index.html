How It Works

TheBullpenBet runs machine-learning models against every MLB game and publishes the output here. This page explains what's happening under the hood — the inputs, the math, and what to trust.

The daily pipeline

Everything you see is produced by a single Python entry point, daily_update.py, that runs three times a day on GitHub Actions: 3 AM, 11 AM, and 5 PM ET. Each run executes ten independent steps; any one failing doesn't block the others, and every artifact it produces is committed back to the repo so the live site (deployed on Render) auto-redeploys with fresh data.

  1. Refresh 2026 Statcast data via pybaseball — pulls every pitch from 2026-03-01 through today incrementally, then rebuilds the four per-game feature CSVs in data/.
  2. Backfill missing game-pick rows — re-runs the game model on any completed date the cron hasn't yet logged.
  3. Run today's game model — projects win probability for every scheduled game, computes EV against current DraftKings/FanDuel lines, flags positive-EV picks.
  4. Re-grade the season pick log — pulls finals from the MLB Stats API and updates the correct column on every prior pick.
  5. Backfill NRFI predictions for past dates — for any past date without an NRFI entry in the picks log, runs the NRFI classifier using only data that was available before that date. Idempotent: already-logged dates are skipped.
  6. Generate today's NRFI predictions — scores every scheduled game through the NRFI classifier and appends the YES/NO lean to 2026_nrfi_picks.csv.
  7. Grade NRFI picks — fetches first-inning linescores from the MLB Stats API for every completed game and writes 2026_nrfi_accuracy.csv.
  8. Backfill missing past-date player snapshots — for any past date without a snapshot on disk, fetches the actual lineup from MLB box scores and runs the player projection models against it. Idempotent: dates that already have a populated snapshot are skipped.
  9. Generate today's hitter and pitcher projections — writes both the live hitterspitchers_today.csv the front-end reads and a dated snapshot for tonight's grading.
  10. Grade player snapshots and recompute calibration — joins every snapshot to its game's actual box score, rebuilds 2026_player_accuracy.csv, then re-derives the bias-correction block applied to today's live projection.

Where the data comes from

Every projection is built from live 2026 Statcast pitch-level data. The data pipeline:

  • Pulls fresh Statcast pitches via pybaseball.statcast(). The cache (pitch_data_2026.csv) lives in the GitHub Actions cache between runs and is updated incrementally — only new dates get fetched.
  • Aggregates each pitch into per-batter and per-pitcher game rows: PA, H, HR, BB, K, plus Statcast metrics like average exit velocity, max EV, average launch angle, barrel rate, and hard-hit proxy.
  • Computes rolling features for every player at five window sizes — last 3, 7, 10, 20, and 50 games — plus career-to-date and standard-deviation columns. These are the same windows the trained models expect at inference time.
  • Computes vs-handedness splits separately so a hitter's "h_rate vs LHP" and "vs RHP" are tracked as distinct features — same for pitchers vs LHB / RHB.
  • Builds opposing-team and opposing-starter context columns so a hitter's projection is conditioned on who they're actually facing.
  • Pulls live odds from The Odds API (DraftKings + FanDuel) and probable starters from MLB Stats API just before each cron tick.

The output is four tables in data/ — hitter_game_data.csv, pitcher_game_data.csv, team_batting_hand_context.csv, team_pitching_hand_context.csv — that every model reads on every run. The grader filters all rows to game_date < target_date at projection time, so the model never sees information that wasn't available before the game started.

Game predictions

The model that produces today's picks is a gradient-boosted classifier (XGBoost) trained on the full 2025 season as the labeled history. For each 2026 matchup it scores live features built from the day's freshly-refreshed Statcast tables:

  • Both teams' last-10-game offensive stats (OPS, K-rate, BB-rate, runs scored, runs allowed)
  • Each starting pitcher's last-10-game and season FIP, K-BB%, and HR/9
  • Both bullpens' season + last-14-day ERA proxy, WHIP, and K-BB%
  • Season-long team OPS, team K-rate, run differential, and win percentage
  • A home-field flag

The model returns a probability that the home team wins — for example, 57.3%. Whichever team has the higher probability becomes the model pick. The probabilities you see in win-prob bars and pick percentages come straight from this output.

Expected value (EV)

Win probability alone doesn't tell you whether a bet is worthwhile — the odds have to support it. EV is the math that connects the two: 

EV = win_prob × (decimal_odds − 1) − (1 − win_prob)

In plain English: if the model thinks BOS wins 57% of the time and the moneyline is +110 (1.91 in decimal), the bet pays out 91¢ per $1 staked when it wins and loses $1 when it loses. Multiply each outcome by its probability and net them. Anything > 0 is positive EV — the model thinks the line is mispriced in your favor.

A pick is flagged Good Value when EV is at least +2% (a 0.02 floor in our config). All others are Pass. Both DraftKings and FanDuel lines are pulled and the better of the two is used for each side — the same edge you'd get by line-shopping yourself.

NRFI predictions

The NRFI (No Run First Inning) model predicts whether neither team will score in the opening inning — a popular betting market. It's a binary classifier that outputs P(NRFI) for each game, trained on every 2026 game in the Statcast database with a labeled first-inning outcome.

Feature priority is intentional. Full-game pitcher stats are the strongest signal because they're built from 80–120 batters per start — far more stable than first-inning-specific rates (which come from 3 batters). The model uses four feature tiers:

  • Full-game pitcher stats (primary) — each starter's rolling K%, BB%, H%, and HR% over the last 5 and 10 games, plus season-to-date standard deviation and days of rest.
  • Full-game team offense (secondary) — rolling team K%, OBP, and runs per game over the last 5 and 10 games for both lineups.
  • First-inning pitcher rates (tertiary) — per-PA K, BB, and runs-allowed rates in the 1st inning specifically — noisy but inning-specific.
  • First-inning team batting (tertiary) — on-base rate, K rate, and runs scored in the 1st inning.

Four model types are trained each run (L1 logistic regression, gradient boosting, random forest, XGBoost) and the one with the best hold-out AUC is selected. Logistic regression with heavy L1 regularization is the default baseline — it auto-selects the most reliable features and is least prone to overfitting the many correlated rolling windows. A confidence threshold is tuned on the hold-out set before publishing predictions.

The lean shown on the NRFI tab is YES (expect a scoreless first) or NO (expect at least one run). Historical accuracy is tracked in 2026_nrfi_accuracy.csv and graded after every game via the MLB Stats API first-inning linescore endpoint.

Player projections

Pitcher and hitter projections come from independent models, one per stat. Each is a regularized XGBoost regressor trained to predict counting stats directly — not rates — using features pulled from the daily-refreshed 2026 Statcast tables. The trained model files live in models/ and are committed to the repo on every training run.

Pitchers

The pitcher pipeline runs five regressors and combines them into the projection you see on the site:

  • pitcher_IP.pkl — projected innings pitched
  • pitcher_K_rate.pkl — strikeouts per batter faced
  • pitcher_BB_rate.pkl — walks per batter faced
  • pitcher_H_rate.pkl — hits allowed per batter faced
  • pitcher_HR_rate.pkl — home runs allowed per batter faced
  • pitcher_runs_per_inning.pkl — earned runs per inning, used to derive projected ER

Per-batter rates are multiplied by expected batters faced (derived from projected IP) to get projected counts. Park factors and hand splits adjust for matchup context. Velocity, spin rate, and pitch-mix percentages from the 2026 Statcast cache feed in as additional features. Models use deep regularization (max_depth=3, min_child_weight=30) to prevent hot-streak overfitting.

Hitters

The hitter pipeline runs seven regressors:

  • hitter_PA.pkl — projected plate appearances (driven by lineup spot + game length features)
  • hitter_H.pkl, hitter_HR.pkl, hitter_BB.pkl, hitter_K.pkl — projected counting stats (hits, home runs, walks, strikeouts)
  • hitter_r_rate.pkl, hitter_rbi_rate.pkl — runs scored and RBI per PA

Hitter projections are blended 75/25 toward the player's season-to-date average before being passed through hard caps — this regression-to-mean shrinkage prevents one hot week from inflating projections to implausible levels. Each hitter's projection is also conditioned on the opposing starter (their 2026 K%, H%, and HR% against same-handed batters), so the same hitter shows different numbers depending on who's pitching. FanDuel player prop lines are displayed alongside projections for direct comparison.

Pitcher and hitter projections are estimated independently and may not sum perfectly across a matchup. Each is a standalone estimate — best used to compare players against each other, not to add up.

The 20-80 confidence grade

Borrowed from MLB scouting reports, the 20-80 scale tells you how much to trust an individual projection. 50 is league average. Higher means the model has more reliable data to work with.

80Elite — established veteran with deep history matching today's situation 70Plus-plus — long track record, well-modeled matchup 65Strong-plus — extensive data, well-supported matchup 60Plus — established player, reliable matchup data 55Slightly above average reliability 50Average — typical projection with some unknowns 45Slightly below average — limited recent data 40Below average — projecting from team averages more than the player's own history 35Sparse data — fallback driven, few prior matchups 30Poor — call-up or substitute, projection leans on league means 20No data — pure league-average estimate, take with a grain of salt

Following MLB scouting convention, the scale skips 25 and 75 — values jump straight from 20 to 30 and from 70 to 80 at the extremes, with the half-grades (35, 45, 55, 65) filling in the middle.

For pitchers, the grade is driven by appearance count plus IP consistency. A starter with 30+ prior outings and stable inning workloads grades 70+; a midseason call-up with 4 starts grades in the 30s.

For hitters, the grade factors in lineup status (Confirmed beats Expected beats fallback), whether starter context was successfully applied, and how much per-game history exists for that player.

Importantly, the grade is independent of the projection itself. A 60-grade pitcher with a poor projection means "we're confident he'll pitch poorly tonight." A 30-grade hitter with a great projection means "the model thinks he'll do well, but we don't have enough on him to bet the house."

How picks are graded

Once a game is final, we pull the box score from the MLB Stats API and compare the model's pick to the actual winner:

  • actual_winner = the team that scored more runs
  • correct = 1 if the model's pick matches the actual winner, 0 if it doesn't, blank if the game hasn't finished yet

Picks flagged Good Value at publish time get assigned a 1-unit bet for accounting. Profit and loss are calculated against the moneyline that was published when the pick was made — not the closing line:

  • Win on +120 → +1.20 units
  • Win on -150 → +0.67 units
  • Loss → -1.00 unit
  • Pass (not bet) → 0.00 units

Running ROI is cumulative profit divided by total units wagered. The Season Accuracy tab shows the cumulative pick accuracy across all graded games — including the ones flagged Pass — so the chart reflects the model's overall predictive performance, not just bet outcomes.

Player projection accuracy

Game-pick accuracy is binary — the predicted winner either won or didn't. Player projections are continuous, so they're scored differently. Every day's hitter and pitcher projections are saved as a dated snapshot in outputs/hitterspitchers_<date>.csv; once the game completes, those projections are joined to the actual MLB box score (pulled from the MLB Stats API) and we record the per-stat difference in 2026_player_accuracy.csv.

For dates earlier in the season that don't yet have a snapshot on disk (typical for the first cron run after deploying), the pipeline backfills them by pulling the actual lineup that played from the box score and running the projection model with target_date set to that game's date. The model only ever sees rows where game_date < target_date, so backfilled projections use the same information the model would have had if it had been run live on that day — no future-data leakage.

Because the data refresh, projection, and grading all happen inside the same cron tick, the projections being graded each day are produced from the same 2026 Statcast features that today's projections use. The grader is fully idempotent: re-running it pulls fresh box scores for any game that's now complete and overwrites the log.

Bias calibration

When the graded log shows the model has a consistent bias on a stat — for example, projecting an average of 3.5 plate appearances when hitters actually average 4.0 — we apply a post-hoc additive correction at projection time so the bias-corrected number is what users see. calibrate_projections.py runs at the end of every cron tick and:

  1. Reads 2026_player_accuracy.csv and computes the average proj − actual for each calibratable column.
  2. Writes the result to calibration.json.
  3. Applies the corrections in-place to outputs/hitterspitchers_today.csv.

Corrections are conservative — they only fire when there are at least 30 graded games for that stat AND the absolute bias is at least 0.05. Counting stats are clipped at zero after the shift so a calibrated projected hits never goes negative.

The biggest leverage points are PA for hitters and IP for pitchers — every other counting stat is derived by multiplying a per-PA / per-batter rate by these, so calibrating those two cascades to fix hits, HR, K, BB, and the rest. The "Live bias calibration" card on the Player Predictions tab shows the current correction for every projection column, with a ✓ for stats currently being corrected and a · for stats that don't yet have enough graded games.

Calibration is a stop-gap that closes the bias gap while we accumulate enough 2026 graded data to retrain the underlying models on. Once retrained, the observed biases should approach zero and the calibration card will show fewer ✓ marks over time.

The Season Accuracy tab's Player Predictions sub-tab summarizes those graded projections in three ways:

  • MAE (mean absolute error) — the average distance from a projection to the actual outcome. Lower is better. For pitcher strikeouts, an MAE of 1.4 means the typical projection lands within ~1.4 K of the real number.
  • Bias — the average signed error (proj − actual). A positive bias means the model over-projects; a negative bias means it under-projects.
  • Within ±tolerance — the share of projections that landed inside a stat-specific tolerance band (e.g. ±1.5 K for pitchers, ±0.5 HR for hitters). This is the most intuitive number for non-stats readers.

Because player projections are graded fresh against MLB Stats API box scores after every slate, the player accuracy view fills in over time as more games are played.

What this site is, and isn't

This is research and analytics, not advice. The model is right around 60% of the time on graded picks — meaningfully better than 50/50, but no projection model wins every day, every series, or every season. Edges in baseball are small and noisy, and the models here are no exception.

Use these picks as one input among several. Cross-check against your own knowledge of the matchup, weather, late-breaking lineup news, and bullpen availability. Bet only what you can afford to lose.
